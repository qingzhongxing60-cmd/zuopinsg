# 模块依赖关系描述规范

> 供 req-analyzer agent 使用。分析功能模块时，必须识别并输出模块间的依赖关系。

---

## 依赖关系的三种类型

### 类型一：数据来源依赖

本模块的某个字段，数据来源于另一个模块。

**识别方式：** 表单中有"选择"类字段（下拉、层级选择、多选），其选项来自其他模块的数据。

**输出格式：**
```
[本模块].[字段名] 来源于 [依赖模块]
  - 依赖条件：[依赖模块]中[状态/条件]的数据
  - 影响：[依赖模块]无数据时，本模块该字段无可选项
```

**示例：**
```
设备档案.设备分类 来源于 设备分类管理
  - 依赖条件：设备分类中已启用的分类
  - 影响：设备分类为空时，新增设备时分类字段无可选项

设备档案.所属单位 来源于 组织管理.部门管理
  - 依赖条件：部门管理中已启用的部门
  - 影响：无部门数据时，所属单位字段无可选项
```

---

### 类型二：状态联动影响

本模块的数据状态变更，影响其他模块的行为。

**识别方式：** 业务规则中有"停用后不可选"、"删除后影响"、"状态变更后"等描述。

**输出格式：**
```
[本模块].[状态变更] → 影响 [目标模块].[具体行为]
  - 触发条件：[状态从X变为Y]
  - 影响范围：[目标模块中哪些操作受影响]
  - 影响方式：[不可选/隐藏/自动更新等]
```

**示例：**
```
设备档案.设备状态=停用 → 影响 巡检计划.检查点选择
  - 触发条件：设备状态变更为"停用"
  - 影响范围：巡检计划配置路线时，该设备对应的检查点
  - 影响方式：检查点不可选

设备分类.状态=停用 → 影响 设备档案.设备分类字段
  - 触发条件：设备分类被停用
  - 影响范围：新增/编辑设备时的分类选择
  - 影响方式：停用的分类不出现在选项中
```

---

### 类型三：关联保护约束

本模块的数据被其他模块引用时，不能直接删除。

**识别方式：** 业务规则中有"删除前检查"、"有关联时阻止"、"先解除绑定"等描述。

**输出格式：**
```
[本模块] 被 [引用模块] 引用
  - 引用关系：[引用模块]的[字段名]引用了[本模块]的数据
  - 保护规则：[本模块]有被引用数据时，阻止删除
  - 提示文案：[原文]
```

**示例：**
```
设备分类 被 设备档案 引用
  - 引用关系：设备档案.设备分类 引用了 设备分类 的数据
  - 保护规则：该分类下有设备时，阻止删除分类
  - 提示文案：该分类下有 X 台设备，请先将设备调整至其他分类

巡检路线 被 巡检计划 引用
  - 引用关系：巡检计划.执行路线 引用了 巡检路线 的数据
  - 保护规则：路线被计划引用时，阻止删除路线
  - 提示文案：该路线已被巡检计划引用，请先解除关联
```

---

## req-analyzer 输出格式

分析完一个功能模块后，在摘要末尾附上依赖关系：

```
## 模块依赖关系

### 数据来源依赖
- [字段名] ← [依赖模块]（[依赖条件]）
- [字段名] ← [依赖模块]（[依赖条件]）

### 状态联动影响
- [本模块状态变更] → [目标模块受影响行为]

### 关联保护约束
- 本模块被 [引用模块] 引用，有引用时阻止删除
- 本模块引用了 [被引用模块]，需先确认被引用模块有数据
```

## 初始化顺序推断

根据依赖关系，推断功能模块的初始化顺序（哪些模块需要先配置）：

```
基础数据层（无依赖）：
  设备分类、部门管理、岗位管理、问题类型

业务数据层（依赖基础数据）：
  设备档案（依赖：设备分类、部门、厂商）
  人员管理（依赖：部门、岗位、角色）
  巡检点（依赖：部门、设备档案）

业务流程层（依赖业务数据）：
  巡检路线（依赖：巡检点）
  巡检计划（依赖：巡检路线、人员）
  巡检任务（依赖：巡检计划）
  整改工单（依赖：巡检任务、人员、问题类型）
```
