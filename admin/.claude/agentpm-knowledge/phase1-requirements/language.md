# 可行性研究报告语言规范

> 供 feasibility-writer agent 使用。撰写每个章节前必须对照本规范检查。

---

## 核心原则

**可研报告是给决策层和审批方看的，语言必须正式、客观、可量化。**

每写一句话前必须自问：
- 这句话是"描述业务价值"还是"描述技术实现"？（可研报告写业务价值）
- 这个数据有依据吗？（不能凭空捏造，无依据时用行业参考值并注明）
- 审批方能看懂吗？（不需要技术背景）

---

## 禁止词汇替换表

| 禁止写法 | 正确替换 |
|---------|---------|
| 弹窗、Modal、Drawer | 弹出窗口、操作界面 |
| a-xxx、el-xxx、van-xxx | 禁止出现任何 UI 组件名 |
| flex、grid、margin、padding | 禁止出现任何 CSS 属性 |
| API、接口路径、mock、axios | 禁止出现任何技术实现词 |
| 路由、router、组件、component | 禁止出现任何前端架构概念 |
| 数据库查询、SQL | 系统检索、系统查找 |
| 前端过滤 | 在已加载数据中过滤 |
| 调用接口 | 提交、查询、获取 |
| 持久化、缓存 | 保存、记录 |
| 我们、我方 | 本项目、建设单位 |
| 很好、非常、极大 | 用具体数字替代（如"提升30%"） |

---

## 四类禁止内容

| 类型 | 特征 | 示例 | 正确写法 |
|------|------|------|---------|
| UI 组件名 | 具体组件名称 | "使用 ECharts 折线图" | "以折线图形式展示趋势" |
| 接口路径 | /api/xxx 格式 | "调用 /api/workorder/list" | "查询整改工单台账" |
| CSS 属性 | 样式描述 | "蓝色按钮，圆角4px" | 删除，不在可研中描述样式 |
| 口语化表达 | 非正式用语 | "这个功能很好用" | "该功能可有效提升作业效率" |

---

## 数字与单位规范

- 金额统一用"万元"为单位，保留两位小数（如：65.00万元）
- 时间节点用"T+N月"表示（T为可研报告批复时间），不写具体日期
- 百分比保留整数（如：99.5%）
- 并发用户数写"不低于N人"
- 响应时间写"不超过Nms/秒"

---

## 章节编号规范

可研报告章节使用中文数字编号：

```
一、现状及必要性
二、建设目标及内容
三、技术方案
四、投资估算
五、项目组织及计划安排
六、可行性分析
```

小节使用阿拉伯数字：1.1、1.2、2.1、2.2 等。

**禁止使用"第1章"、"第一章"、"Chapter 1"等格式。**

---

## 不适用章节处理规范

不适用的章节**必须保留标题**，正文写明原因，不能完全跳过：

```markdown
### 1.3 系统现状

本项目为自主研发类项目，建设单位当前未部署同类信息化系统，本节不适用。
```

**禁止直接删除不适用章节，否则章节编号会断裂。**

---

## 封面格式规范

封面必须使用 YAML frontmatter 格式：

```yaml
---
title: "可行性研究报告"
subtitle: "{项目名称}"
author: "{客户单位名称}"
date: "编制单位：{编制单位名称}\n\n{年月中文，如：二零二六年五月}"
lang: zh-CN
toc: true
toc-depth: 3
numbersections: false
geometry: "left=3.17cm,right=3.17cm,top=2.54cm,bottom=2.54cm"
---
```

**禁止使用 `<br/>`、`<br>` 等 HTML 标签。**
