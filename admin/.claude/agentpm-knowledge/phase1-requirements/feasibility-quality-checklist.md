# 可行性研究报告质量审查清单

> 供 feasibility-reviewer agent 使用。逐项检查，每项必须明确标注 ✅ 通过 / ❌ 不通过（含具体问题位置）。
> **禁止笼统说"全部通过"，每项必须逐一标注结果。**

---

## 维度一：结构完整性

### 1.1 章节编号格式

- [ ] 一级章节使用中文数字（一、二、三、四、五、六）
- [ ] 小节使用阿拉伯数字（1.1、1.2、2.1 等）
- [ ] 无"第N章"、"Chapter N"等错误格式

### 1.2 对照模板检查各章节必要小节

**一、现状及必要性**
- [ ] 1.1 项目背景：存在，300-500字
- [ ] 1.2 国内外现状及发展趋势：自主研发类/引进实施类必须有
- [ ] 1.3 系统现状：提升推广类必须有；其他类型保留标题写"本节不适用"
- [ ] 1.4 主流产品分析：引进实施类必须有；其他类型保留标题写"本节不适用"
- [ ] 1.5 需求和必要性分析：存在，包含业务需求分析和必要性分析

**二、建设目标及内容**
- [ ] 2.1 顶层设计及总体规划：存在
- [ ] 2.2.1 业务及功能目标：存在，至少3条
- [ ] 2.2.2 系统性能目标：存在，有表格，含响应时间/并发/可用性指标
- [ ] 2.2.3 系统运维目标：存在
- [ ] 2.3 项目开发功能：存在，有功能清单表格（含"建设类型"列）
- [ ] 2.4 项目实施内容：存在，有具体实施步骤
- [ ] 2.5 项目运维方案：存在，含备份/监控/故障处理

**三、技术方案**
- [ ] 3.1.1-3.1.8 全部小节存在
- [ ] 3.1.4 技术架构：有前端和后端技术栈表格
- [ ] 3.1.7 性能指标：有表格，数据与2.2.2一致
- [ ] 3.2-3.5 全部存在

**四、投资估算**
- [ ] 4.1 汇总表格：8个费用类别全部列出，合计正确
- [ ] 4.2.1-4.2.8 全部小节存在（不适用的写"本项目不涉及"）
- [ ] 4.3 投资分配表：按阶段分配，时间用T+N月格式

**五、项目组织及计划安排**
- [ ] 5.1 项目组织：有组织架构描述和职责表格
- [ ] 5.2 里程碑计划：有表格，时间用T+N月格式

**六、可行性分析**
- [ ] 6.1 技术论证：存在，有明确可行性结论
- [ ] 6.2 效益分析：存在，含经济效益（有量化数据）和社会效益
- [ ] 6.3 风险评估：存在，有风险表格（含风险等级和规避措施）

---

## 维度二：数据一致性

- [ ] 项目名称在全文统一（与封面一致）
- [ ] 编制单位在全文统一
- [ ] 投资总额：4.1汇总 = 4.2各项之和 = 4.3分配表合计
- [ ] 性能指标：2.2.2 与 3.1.7 数据一致
- [ ] 建设周期：5.2里程碑总时长与封面/正文描述一致
- [ ] 功能模块：2.3功能清单与正文描述的模块数量一致
- [ ] 并发用户数：全文统一（不能一处写100人，另一处写200人）

---

## 维度三：语言规范

- [ ] 无 UI 组件名（Modal、Drawer、a-xxx、el-xxx 等）
- [ ] 无接口路径（/api/xxx 格式）
- [ ] 无 CSS 属性（flex、grid、margin、px 等）
- [ ] 无口语化表达（"我们"、"很好"、"非常"等）
- [ ] 金额单位统一为"万元"，保留两位小数
- [ ] 时间节点使用"T+N月"格式，不写具体日期
- [ ] 无 `<br/>`、`<br>` 等 HTML 标签
- [ ] 章节编号格式正确（中文数字一级章节）

---

## 维度四：可研报告要求

- [ ] 封面使用 YAML frontmatter 格式
- [ ] 文档信息表存在（含历史版本表）
- [ ] 不适用章节保留标题并写明原因（非直接删除）
- [ ] 投资估算有估算依据说明（人月单价、工作量等）
- [ ] 里程碑计划有交付物说明
- [ ] 风险评估有风险等级（高/中/低）
- [ ] 可行性分析有明确结论（"技术可行性结论为可行"等）
- [ ] 效益分析有量化数据（节省人工成本N万元/年等）
