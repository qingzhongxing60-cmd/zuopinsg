# PRD 语言规范

## 核心原则

**只写"用户能感知到的功能行为"，不写"实现这个行为的技术手段"。**

每写一句话前必须自问：
- 产品经理/测试/业务方能看懂吗？（不需要技术背景）
- 这句话描述的是"做什么"还是"怎么做"？（只写做什么）
- 换一种技术栈，这句话还成立吗？（与实现无关）

---

## 禁止词汇替换表

| 禁止写法 | 正确替换 |
|---------|---------|
| 弹窗、Modal、Drawer、抽屉 | 弹出编辑窗口、弹出确认、右侧编辑面板 |
| 步骤条、Steps 组件 | 分步填写（共N步） |
| Tab 页签、Tabs | 分区展示、切换查看 |
| Tag 形式展示、标签组件 | 逐个展示，支持移除 |
| 联动控件、Radio、Checkbox | 单选、多选 |
| 树形下拉、TreeSelect | 层级选择 |
| hover 显示、tooltip | 悬停显示 |
| 开关组件、Switch | 开关、状态切换 |
| 回填、回显 | 自动填入已有数据 |
| 工具栏、操作栏 | 直接描述功能位置 |
| a-xxx、el-xxx、van-xxx | 禁止出现任何 UI 组件名 |
| flex、grid、margin、padding | 禁止出现任何 CSS 属性 |
| API、接口、mock、axios | 禁止出现任何技术实现词 |
| 路由、router、组件、component | 禁止出现任何前端架构概念 |
| 数据源：/api/xxx | 来源于 [业务模块名] 中已启用的 [数据名] |
| 前端过滤、不请求接口 | 在已加载数据中过滤 |
| 调用接口、API 请求 | 提交、查询、获取 |
| 数据库查询 | 系统查找、系统检索 |
| 持久化、缓存 | 保存、记录 |

---

## 四类禁止内容

| 类型 | 特征 | 示例 | 正确写法 |
|------|------|------|---------|
| 视觉设计 | 颜色、字号、间距、圆角 | "按钮蓝色，圆角4px" | 删除，或改为"主要操作按钮" |
| 布局设计 | 位置、排列、对齐、宽高 | "左右布局，左侧占30%" | 删除，或改为"分区展示" |
| 交互细节 | 动画、过渡、悬停效果 | "淡入动画，hover显示" | 改为"悬停显示提示信息" |
| 技术实现 | 组件名、API、数据结构 | "使用TreeSelect组件" | 改为"层级选择" |
| 实现说明 | 解释为什么这样设计、跨章节引用、开发注意事项 | "> 说明：此功能仅在移动端实现" | 删除，需求说明书只描述功能本身，不写实现原因和注释 |

## 数据来源的正确写法

描述字段的数据来源时，用业务语言说明来源范围，不写接口路径：

```
❌ 数据源：/department/tree，支持搜索
✅ 来源于组织管理中已启用的部门，支持按名称搜索

❌ 数据源：厂商信息列表
✅ 来源于厂商管理中已录入的厂商

❌ 数据源：设备分类树
✅ 来源于设备分类中已启用的分类，按层级展示

❌ 数据源：启用状态的巡检路线列表
✅ 选择范围为已启用的巡检路线
```

---

## 正确的 PRD 三要素

每个功能描述必须包含：

1. **数据定义**：展示/操作什么数据，数据来源和含义
2. **业务规则**：什么条件下显示/隐藏，数据如何计算/判断
3. **交互功能**：用户能做什么操作，操作后发生什么

缺少任何一项都是不完整的需求。

---

## 业务术语一致性

同一项目中，相同概念必须使用统一术语，不能混用：

| 统一用法 | 禁止混用 |
|---------|---------|
| 启用/停用 | 不能和"正常/禁用"、"开启/关闭"混用 |
| 新增 | 不能和"添加"、"创建"混用 |
| 编辑 | 不能和"修改"、"更新"混用 |
| 删除 | 不能和"移除"、"清除"混用 |

> req-writer 在写作前必须读取项目记忆中的业务术语表，保持一致。

## 内容分级规则

需求文档中不同章节对内容粒度有不同要求，分为三个层级：

### L1 概览层（3.2 需求功能清单、3.3 页面功能清单）

只写"用户能做什么"，格式：动词 + 对象 + 条件（可选）

✅ "按油库、油品两个维度查看占比统计"
✅ "查看总运单数、总接卸量、完成率、单均接卸量四项指标"
❌ "4 张 KPI 卡，顶部 3px 彩色细条做色彩归类"
❌ "折线 + 蓝色面积渐变图"

**L1 禁止出现：**
- 图表视觉类型描述（折线图→趋势图，柱状图→排名图，饼图→占比图，圆环饼图→占比图）
- 颜色描述（蓝色渐变、紫色渐变、彩色细条 → 删除）
- 尺寸数值（px、rem、% → 删除）
- 布局位置（左侧、右侧、顶部、底部 → 删除）
- 组件数量+视觉描述组合（"4 张 KPI 卡" → "四项指标"）

### L2 行为层（3.5.x 功能设计主体）

写"功能做什么、数据是什么、规则是什么"，不写"长什么样"。

✅ "展示 4 项 KPI 指标：总运单数、总接卸量、完成率、单均接卸量"
✅ "支持按日/周/月切换时间维度"
❌ "顶部 3px 彩色细条" / "毛玻璃效果，宽 320px"
❌ "cubic-bezier(0.4, 0, 0.2, 1) 动画过渡"

**L2 禁止出现：**
- 像素值（Npx、Nrem）
- 颜色代码（#xxx、rgb、rgba）
- CSS 属性名（border-radius、box-shadow、backdrop-filter）
- 动画描述（cubic-bezier、淡入、滑出、ease-in、ease-out）
- 字体具体数值（字号 Npx、font-weight）
- 视觉效果词（毛玻璃、渐变、阴影）

### L3 规格层（「视觉与交互规格」独立子节，可选）

不受 L1/L2 禁止词约束，可写具体视觉实现细节。
物理独立于功能行为描述，用独立小标题包裹。
仅在功能有特殊视觉要求、需要明确传达给设计/开发时才添加。
