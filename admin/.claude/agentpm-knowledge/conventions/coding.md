# 编码规范

## 克制原则

**只做被要求的事，不要自作主张添加额外内容。**

- 让你去掉边框，就只去掉边框，不要顺手加阴影、加圆角、加背景色
- 让你加一个按钮，就只加按钮，不要顺手加 tooltip、加动画、加 loading 状态
- 让你改颜色，就只改颜色，不要顺手调整间距或字体
- 需求里没有提到的视觉效果、交互效果、样式属性，一律不加
- 如果你认为某个额外改动有必要，先问用户，不要直接做

---

## 注释规范

**Vue 组件（.vue）**
- `<script setup>` 顶部写组件用途说明（1-2行）
- 每个功能块（数据定义、API调用、事件处理等）前加单行注释说明用途
- 复杂逻辑（如数据转换、条件判断）行内加注释解释原因

```ts
// 表格列定义
const columns = [...]

// 获取列表数据（带分页和筛选）
const fetchList = async () => { ... }

// 删除前检查关联关系，有关联时阻止删除
const handleDelete = async (record) => { ... }
```

**TypeScript/JavaScript（.ts/.js）**
- 每个导出函数/接口前写 JSDoc 注释说明用途和参数
- mock 文件每个接口前注释说明接口路径和用途

```ts
/** 获取角色列表，支持分页和按名称/状态筛选 */
export const getRoleList = (params: RoleListParams) => { ... }
```

**CSS/样式**
- 非显而易见的样式值加注释说明原因

```css
/* 撑满内容区高度，配合 PageContainer flex 布局 */
.table-card { flex: 1; }
```

## 大块内容写入规范

**当需要写入或替换超过 80 行的内容时，禁止一次性使用 Edit/Write 工具，必须分块处理。**

**新建文件（Write）：** 先写前 50 行，剩余内容用多次 Edit 追加，每次不超过 50 行。

**替换已有内容（Edit）：** 使用占位符接力法：
1. 第一次 Edit：替换原内容开头，new_string 末尾加唯一占位符 `<!-- PART_2 -->`
2. 第二次 Edit：把 `<!-- PART_2 -->` 替换为下一段内容，末尾加 `<!-- PART_3 -->`
3. 依此类推，最后一次 Edit 同时把原内容的剩余旧文本也包含在 old_string 里一并清除

---

## 大文件读取规范

**禁止一次性 Read 超过 200 行的文件，必须分段读取。**

- 先用 `Bash grep -n` 定位目标章节/函数/区域的行号
- 再用 `Read` 的 `offset` + `limit` 参数只读取需要的部分
- 需要了解文件整体结构时，用 `grep -n "^##\|^###\|^export\|^const\|^function"` 提取骨架

---

## 并发操作规范

**同一条消息内，所有无依赖关系的操作必须并行执行，不能串行。**

- 多个文件读取 → 同一条消息并行发出
- 多个文件写入/编辑 → 同一条消息并行发出
- 多个 Bash 命令（无依赖关系）→ 同一条消息并行发出
- 只有当 B 依赖 A 的输出结果时，才允许串行

---

## 文件规模规范

- 单个文件超过 500 行时，考虑拆分为多个文件，每个文件职责单一
- 拆分原则：按功能模块拆分，而非按技术层级拆分

---

## 系统边界校验

- 用户输入、外部 API 返回值、URL 参数等外部数据，必须在使用前校验
- 不信任系统边界之外的任何数据，内部模块间调用无需重复校验

---

## 硬性规则（必须遵守）

以下规则不可绕过，生成代码时必须严格执行：

1. **单文件行数** ≤ 500 行；单个函数/方法 ≤ 80 行
2. **禁止 `console.log`**，调试日志必须在提交前清除
3. **禁止裸写 `fetch` / `XMLHttpRequest`**，统一通过项目封装的 `request` 工具（admin: `@/utils/http`，mobile: `@/utils/request.js`）
4. **禁止在组件内直接操作 DOM**，使用 Vue 响应式或 `ref` / `useTemplateRef`
5. **Mock 数据变量必须在模块顶层声明**，不能写在 handler 函数内部（否则增删改不生效）
6. **新增页面必须同步添加路由**，不允许存在无路由的孤立页面
7. **API 函数必须有 JSDoc 注释**，说明接口路径和用途

---

## 配置保护规则

**禁止通过修改配置来消除代码报错。修代码，不改规则。**

以下文件未经用户明确许可不得修改：
- `.eslintrc.*` / `eslint.config.*` — 不能添加 disable 规则
- `.prettierrc.*` — 不能放宽格式要求
- `tsconfig.json` — 不能放宽类型检查（如 `strict: false`、`skipLibCheck: true`）
- `vite.config.*` — 不能关闭类型检查插件

遇到 lint/type 报错时的正确做法：
1. 修复代码使其符合规则
2. 如果规则确实不合理，向用户说明并请求许可后再改配置

## 提交规范

格式：`type(scope): subject`

| type | 说明 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复缺陷 |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 代码重构（不含 bug 修复和功能新增） |
| `perf` | 性能优化 |
| `test` | 添加或修改测试 |
| `build` | 构建流程、外部依赖变更 |
| `ci` | CI 配置、脚本修改 |
| `revert` | 回滚 commit |
| `chore` | 构建工具或辅助库的更改 |

示例：
```
feat(equipment): 新增设备档案列表页
fix(auth): 修复登录 token 过期未跳转问题
docs(readme): 更新本地开发启动说明
```
