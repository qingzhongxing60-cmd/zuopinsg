# 验证铁律

## 核心原则

```
声称完成 = 必须有验证证据
没有证据的完成声明 = 谎言
```

**违反这条规则的字面意思就是违反这条规则的精神。**

---

## 门控函数

在声称任何状态或表达满意之前：

1. **识别**：什么命令能证明这个声明？
2. **执行**：在当前消息中运行完整命令
3. **读取**：完整输出，检查退出码，计数失败
4. **验证**：输出是否确认了声明？
   - 否 → 陈述实际状态 + 证据
   - 是 → 陈述声明 + 证据
5. **然后才能**：做出声明

跳过任何一步 = 未验证

---

## 常见失败

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| 测试通过 | 测试命令输出：0 failures | 上次运行、"应该通过" |
| Lint 干净 | Lint 输出：0 errors | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Lint 通过、日志看起来正常 |
| Bug 已修复 | 测试原始症状：通过 | 代码改了、假设修好了 |
| 需求已满足 | 逐行对照检查清单 | 测试通过 |
| 子 Agent 完成 | VCS diff 显示变更 | Agent 报告"成功" |

---

## Red Flags — 立即停止

如果你发现自己：
- 使用"应该"、"大概"、"似乎"
- 在验证前表达满意（"好了！"、"完成！"、"搞定！"）
- 准备提交/推送/PR 但没有验证
- 信任子 Agent 的成功报告
- 依赖部分验证
- 想着"就这一次"
- **任何暗示成功但没有运行验证的措辞**

---

## 反借口表

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 运行验证命令 |
| "我很确定" | 确信 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Lint 通过了" | Lint ≠ 编译器 ≠ 运行时 |
| "Agent 说成功了" | 独立验证 |
| "改动很小" | 小改动也能引入类型错误 |
| "部分检查够了" | 部分证明不了什么 |
| "上次验证过了" | 每次变更后都需要重新验证 |

## 关键模式

### 测试验证
```
✅ [运行测试命令] [看到: 34/34 pass] "所有测试通过"
❌ "应该通过了" / "看起来对的"
```

### 构建验证
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Lint 通过了"（Lint 不检查编译）
```

### 需求验证
```
✅ 重读需求 → 创建检查清单 → 逐项验证 → 报告缺口或完成
❌ "测试通过了，功能完成"
```

### 子 Agent 委托验证
```
✅ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任 Agent 报告
```

---

## 何时应用

**始终在以下之前：**
- 任何成功/完成的声明
- 任何满意的表达
- 任何关于工作状态的正面陈述
- 提交、PR 创建、任务标记完成
- 移动到下一个任务
- 委托给子 Agent 后

**这是不可协商的。**
