# 开发工作流

## 功能实现流程

### 第 0 步：研究优先（任何新实现前必须）

在写任何新代码之前：
1. 搜索项目内是否已有类似实现
2. 检查 npm/包注册表是否有成熟的库
3. 优先复用已有代码，而非重新造轮子
4. 找到可以适配的方案时，移植而非从头写

### 第 1 步：先规划

- 明确需求和边界
- 识别依赖和风险
- 拆分为可独立完成的子任务

### 第 2 步：TDD

- 先写测试（参考 `conventions/testing`）
- 写最小实现让测试通过
- 重构，保持测试绿色

### 第 3 步：代码审查

编写完成后自查：
- [ ] 代码可读，命名清晰
- [ ] 函数 < 50 行，文件 < 800 行
- [ ] 无深层嵌套（> 4 层）
- [ ] 错误处理完整
- [ ] 无硬编码值
- [ ] 无不可变性违反

### 第 4 步：提交

- 遵循 `conventions/git` 的提交格式
- 确保 CI 检查通过后再请求 Review

## 验证铁律（不可跳过）

**声称完成前，必须在当前消息中提供验证证据。**

规则：
1. 识别验证命令（type-check、lint、test、curl 等）
2. 在当前消息中运行该命令
3. 读取输出结果
4. 确认通过后才能声称完成

以下理由不能作为跳过验证的借口：
- "改动很小不需要验证" — 小改动也可能引入类型错误
- "我确信这是对的" — 确信不等于证据
- "验证太慢了" — 宁可慢也不能交付有问题的代码
- "上次验证过了" — 每次变更后都需要重新验证

## 系统化调试流程

### 阶段 1：根因调查

1. 读取完整错误信息（不要只看第一行）
2. 复现问题（确认不是偶发）
3. 检查最近的变更（git diff / git log）
4. 在组件边界处收集证据

### 阶段 2：模式分析

1. 找到正常工作的类似代码
2. 对比差异，缩小范围
3. 检查是否是已知问题（搜索 issue）

### 阶段 3：假设与测试

1. 一次只验证一个假设
2. 做最小改动验证
3. 改动无效立即回滚

### 阶段 4：三次失败规则

**如果同一个方向尝试 3 次仍然失败，停止修补，质疑架构。**

- 不要做第 4 次微调
- 退后一步，分析根本原因
- 考虑是否需要换一种方案
- 向用户说明情况，提出替代方案

## 硬边界（绝对禁止）

以下规则不存在例外：

1. **禁止在 `src/` 之外生成业务代码** — 页面、组件、API 只能在项目 src/ 目录内
2. **禁止修改 linter/formatter 配置来消除报错** — 修代码，不改规则
3. **禁止未经阅读就修改文件** — 必须先 Read 再 Edit，不能凭记忆修改
4. **禁止修改 `.claude/knowledge/` 文件** — 除非用户明确要求更新知识库
5. **禁止删除用户未提及的代码** — 即使看起来是"死代码"，也要先确认
6. **禁止在修复 bug 时顺手重构** — 一次只做一件事

## 条件性 Agent 激活

code-reviewer 已整合安全检查，不再需要独立的 security-reviewer。

| 条件 | 激活 Agent |
|------|-----------|
| 修改了 auth/、middleware/、guard/ 下的文件 | code-reviewer（重点审查安全） |
| 新增 API 端点或修改路由 | code-reviewer（重点审查安全） |
| 变更 > 3 个文件 | code-reviewer |
| 涉及数据库 schema 变更 | code-reviewer（重点审查安全） |
| 单文件小修改（< 20 行） | code-reviewer |
| 纯样式/文案修改 | code-reviewer |

## 核心原则

### KISS（保持简单）
- 优先选择最简单的可行方案
- 避免过早优化
- 清晰优于聪明

### DRY（不重复）
- 提取重复逻辑为共享函数
- 避免复制粘贴导致的实现漂移
- 只在重复是真实的时候才引入抽象

### YAGNI（不过度设计）
- 不要构建当前不需要的功能或抽象
- 从简单开始，有真实压力时再重构
- 三处相似代码才考虑抽象，两处不够

## 代码异味（要避免）

- **深层嵌套**：超过 4 层用提前返回代替
- **魔法数字**：用命名常量替代
- **超长函数**：拆分为职责单一的小函数
- **大文件**：按功能/领域拆分，不按类型拆分
