# 功能存在性判定规则

## 唯一真相源：路由注册表

判定一个功能模块是否"存在于项目中"，唯一标准是**路由是否在入口文件中注册**。

| 端 | 路由入口文件 | 判定方式 |
|----|------------|---------|
| Admin | `src/router/modules/index.ts` | 该文件中 import 并导出的路由模块 = 存在 |
| Mobile | `src/router/index.js` 或等效入口 | routes 数组中注册的路由 = 存在 |

## 判定规则

```
路由已注册 → 功能存在
路由未注册 → 功能不存在（无论其他文件是否残留）
```

## 常见残留情况（均视为"功能已删除"）

以下情况都**不代表功能存在**，不应纳入需求说明书、功能清单或任何文档：

- `src/views/Xxx/` 目录存在，但 `router/modules/` 中无对应路由
- `router/modules/xxx.ts` 文件存在，但未在 `index.ts` 中 import/导出
- `src/api/xxx.ts` 存在，但对应路由已删除
- `src/mock/xxx.ts` 存在，但对应路由已删除

## 应用场景

| 场景 | 如何应用 |
|------|---------|
| req-doc 反向同步 | 只将已注册路由对应的功能写入 SRS，残留文件对应的功能从 SRS 中删除 |
| 清理死代码 | 未注册路由对应的 view/api/mock 文件可安全删除 |
| page-generator | 生成新页面时参考已注册路由确定功能范围，不参考残留文件 |
| delivery-plan | 交付计划中的功能列表以路由注册为准 |

## 为什么不以 view 文件为准

开发过程中经常出现：
1. 功能被砍掉，删了路由但忘了删 view 文件
2. 功能重构，旧 view 文件未清理
3. 实验性代码写了 view 但从未注册路由

这些残留文件如果被当作"功能存在"，会导致需求说明书中出现实际不存在的功能描述。
