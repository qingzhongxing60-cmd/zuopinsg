# 交付链路规划器

## 工作模式

| 模式 | 触发场景 | 入口 |
|------|---------|------|
| **生成模式** | 首次为项目生成交付计划 | → Step A |
| **查看模式** | 查看当前进度和推荐 | → Step B |
| **更新模式** | 标记功能完成，更新计划 | → Step C |
| **自动执行模式** | 按计划连续实现所有功能，不停止不询问 | → Step D |

---

## Step A: 生成模式

### A1: 扫描项目上下文

```bash
# 查找需求说明书
find docs/ -name "*需求*说明书*.md" 2>/dev/null
# 查找已有交付计划
ls docs/delivery-plan.md 2>/dev/null
```

- 若已有 `docs/delivery-plan.md`，询问用户是否覆盖重新生成
- 若无需求说明书，提示用户先用 `/req-doc` 生成

### A2: 调用 delivery-analyzer

```
subagent_type: "delivery-analyzer"
传入：
  PROJECT_PATH: {项目根目录绝对路径}
  SRS_PATH: {需求说明书文件路径}
  PROJECT_NAME: {从需求说明书封面提取的项目名称}
```

### A3: 写入计划文件

将 delivery-analyzer 的输出写入 `docs/delivery-plan.md`。

写入后输出：

```
[完成] 交付计划已生成

文件：docs/delivery-plan.md
Phase 1 共 N 个功能模块，分 4 层实现

第1层（N个）：系统基础，无依赖，可立即开始
第2层（N个）：核心业务，依赖第1层
第3层（N个）：衍生功能，依赖第2层
第4层（N个）：移动端，依赖PC端完成

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
推荐第一步：{第1层第一个功能}
执行：/page-generator 实现{功能名}
```

```
依赖状态：{依赖项} 均已完成

```

## Step C: 更新模式

**触发时机**：page-generator 完成验收后自动调用，或用户手动触发。

**触发方式**：
- page-generator Step 7 自动调用（传入已完成的功能名）
- 用户输入"标记{功能名}完成"

**执行步骤**：

1. 读取 `docs/delivery-plan.md`
2. 找到对应功能条目，将状态改为 [完成]，填写完成时间
3. 更新总进度表计数
4. 更新"最后更新"日期
5. 计算下一个推荐功能（依赖已全部满足且状态为 [待实现] 的最小 ID）
6. 更新"当前推荐"章节
7. 写回文件

**输出**：

```
[完成] 已标记完成：{功能名}

Phase 1 进度：{已完成}/{总数}

推荐下一步：{下一个功能}
依赖状态：{依赖项} 均已完成
执行：/page-generator 实现{功能名}
```

## Step D: 自动执行模式

**触发词**："实现全部功能" / "按计划实现所有功能" / "自动实现" / "一直实现" / "连续实现"

**核心原则（来自 ECC-main Infinite Agentic Loop）**：
- delivery-plan.md 是 spec 文件，自动执行模式是 Orchestrator
- 读取计划 → 找下一个可实现功能 → 调用 page-generator 完整实现 → 更新计划 → 继续下一个
- **不停止、不询问、不偷懒**，直到所有依赖满足的功能全部完成
- 遇到依赖未满足的功能才暂停，并说明原因

### D1: 读取计划，确定执行队列

读取 `docs/delivery-plan.md`，按以下规则构建执行队列：

```
1. 默认范围：Phase 1（前端）全部完成后再进入 Phase 2（后端）
   —— 用户可指定 "实现全部后端"/"实现Phase2" 则只构建 Phase 2 队列
2. 遍历目标 Phase 的所有条目
3. 筛选状态为 ⬜ 的条目
4. 检查依赖：依赖列中所有 ID 状态均为 ✅
5. 按层级顺序排列（第1层全部完成后才进入第2层）
6. 同层内按 ID 顺序排列
```

> **技能分发**：每个条目按其"技能"列决定调用哪个技能——
> `page-generator`（前端 F 开头条目）/ `backend-generator`（后端 B 开头条目）。

输出执行队列预览：
```
自动执行队列（共 N 个功能）：
  1. F02 移动端-登录/登出（Phase1 第1层，page-generator）
  2. F08 炼厂管理（Phase1 第2层）
  3. B01 用户管理接口（Phase2 第1层，backend-generator）
  ...

开始执行，不会中途停止询问。
```

### D2: 循环执行

对队列中每个条目，按其"技能"列执行完整流程（page-generator 或 backend-generator，均为步骤1-7）：

```
当前执行：[ID] [功能名]（Phase N 第M层，技能=[page-generator/backend-generator]）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[按技能字段执行对应技能的完整流程]
  - page-generator：需求理解→规范加载→执行计划→代码生成→路由导航→验收检查→更新计划
  - backend-generator：需求理解(读LLD)→规范加载→执行计划→数据模型→生成骨架补业务→验收检查→更新计划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[完成] [功能名] ✅  进度：已完成/总数
继续下一个：[下一个功能名]
```

**执行规则**：
- 按条目"技能"列分发：F 开头 → page-generator，B 开头 → backend-generator
- 每个条目必须完整执行对应技能的全部7个步骤，不能跳步骤
- 验收检查（Step 6）发现问题必须修复后才能继续，不能跳过
- 每个条目完成后立即更新 delivery-plan.md（Step 7）
- 上一个的完成状态会解锁下一个的依赖，自动纳入队列

### D3: 暂停条件

以下情况才暂停并告知用户：
- 所有当前可执行功能（依赖已满足）全部完成
- 遇到需要用户提供信息才能继续的功能（如需要外部 API 密钥等）
- 连续3次验收失败无法自动修复

暂停时输出：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[暂停] 当前可执行功能已全部完成

Phase 1 进度：{已完成}/{总数}
剩余功能等待依赖：
  - F03 移动端-我的（等待 F02 完成）
  - F08 炼厂管理（等待第1层全部完成）

继续执行：/delivery-plan 实现全部功能
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 推荐算法

从计划文件中找"下一步推荐"：

```
1. 遍历 Phase 1 所有条目
2. 筛选状态为 [待实现] 的条目
3. 检查每个条目的依赖：依赖列中所有 ID 的状态均为 [完成]
4. 优先推荐当前层级中满足条件的条目
5. 当前层全部完成后，才推荐下一层的第一个满足条件的条目
6. 同层多个可选时，按 ID 顺序推荐最小的
7. 若 Phase 1 全部完成，提示可以开始 Phase 2
```
