# 概要设计（HLD）内容规范

> 适用于 hld-design 技能与 design-analyzer / design-writer / design-reviewer 三代理。
> 概要设计回答"系统由哪些部分组成、各部分如何协作"，是 SRS 需求到 LLD 实现之间的桥梁。

## 目录

- 一、核心定位
- 二、分层边界（HLD 写什么 / 不写什么）
- 三、各章节内容要求
- 四、与需求文档的语言差异
- 五、质量清单

---

## 一、核心定位

```
SRS（做什么） → HLD（系统怎么搭） → LLD（每块怎么实现）
```

HLD 的读者是架构师、技术负责人、开发组长。HLD 要让他们看完后能回答：
- 系统分几层？每层用什么技术？
- 拆成哪些模块？模块怎么协作？谁依赖谁？
- 核心数据实体有哪些？怎么存？
- 对外/对内有哪些接口分组？
- 怎么部署？怎么满足性能/安全/可用性要求？

**HLD 的每个设计决策必须可追溯到 SRS 的功能需求或非功能需求。** 设计是需求的技术实现，不是再造需求。

## 二、分层边界（HLD 写什么 / 不写什么）

HLD 与 LLD 最容易混淆。判断标准：**HLD 到"模块/实体/接口分组"级即止，再往下的字段、单接口、代码就是 LLD。**

| 维度 | HLD 写（架构级） | 不写（留给 LLD） |
|------|----------------|----------------|
| 模块 | 模块清单、职责、依赖关系 | 模块内部的类、方法实现 |
| 数据 | 核心业务实体、实体间关系、存储选型 | 表结构、字段类型、索引 |
| 接口 | 接口分组、协议、对外/对内边界 | 单个接口的请求/响应/错误码 |
| 流程 | 跨模块的协作流程（概念级） | 单功能内部的处理步骤时序 |
| 技术 | 技术栈选型 + 选型理由 | 框架配置、代码实现 |

**越界示例（HLD 中禁止）：**

| 越界写法（属于 LLD） | HLD 正确写法 |
|--------------------|------------|
| `waybill 表含 plate_no VARCHAR(20)...` | "运单实体，与司机、车辆为多对一关系" |
| `POST /api/waybills 请求体 {plateNo, ...}` | "运单管理接口组，提供增删改查能力" |
| `WaybillService.create() 先校验再入库` | "运单模块负责运单全生命周期管理" |

## 三、各章节内容要求

### 系统总体设计
- 系统定位（一句话说清系统是什么）、目标用户、核心业务能力
- 系统边界：负责什么、不负责什么、与哪些外部系统交互（画系统上下文）
- 必须配**系统总体架构图**（diagram-generator，architecture 类型）

### 应用架构
- **分层设计**：表现层 / 网关层 / 应用服务层 / 领域层 / 数据访问层 / 数据层（按实际裁剪），每层职责 + 关键技术
- **功能模块划分**：模块清单表（模块 / 职责 / 对应SRS功能 / 依赖模块），模块划分依据（按业务域 or 技术分层）
- **模块依赖关系**：依赖方向，避免循环依赖（有环必须说明理由）
- 必须配**应用分层架构图**或**模块关系图**

### 数据架构
- 核心业务实体清单（名词，如"运单""设备"，不是表）
- 实体关系（1:N / N:M），配**概念 ER 图**（diagram-generator，er 类型，概念级不含字段）
- 存储选型：关系库 / 缓存 / 对象存储 / 搜索引擎，各承载什么数据、为何这样划分

### 接口架构
- 接口分组（通常按模块）、对外接口 vs 对内接口边界
- 接口协议：REST / WebSocket / 消息队列 及适用场景
- 第三方集成：短信、地图、支付等外部接口及对接方式

### 技术架构
- 前端 / 后端 / 数据库 / 中间件技术栈，**每项给出选型理由**（来自代码实际依赖或工程权衡）
- 鉴权方案、缓存方案、文件存储方案（架构级，不写配置）

### 部署架构
- 部署形态（单体 / 前后端分离 / 微服务）、环境划分（开发/测试/生产）
- 关键中间件部署、负载/高可用策略（如有）
- 可配**部署架构图**

### 非功能性设计
- 把 SRS 第五章非功能需求逐条映射到架构应对策略：

| SRS非功能需求 | 架构应对策略 |
|-------------|------------|
| 接口响应<1s | 索引优化 + Redis缓存 + 分页 |
| 99.9%可用 | 多实例部署 + 健康检查 |
| RBAC权限 | 网关层统一鉴权 + 权限码控制 |

## 四、与需求文档的语言差异

**HLD 允许、且应该使用技术词汇**——这与 prd-language.md（需求阶段禁止技术词）完全相反。

| 类别 | HLD 中的状态 |
|------|------------|
| 技术栈名（Vue/Spring/MySQL/Redis） | ✅ 必须写 |
| 架构术语（分层/网关/微服务/缓存） | ✅ 必须写 |
| 协议（REST/WebSocket/JWT） | ✅ 必须写 |
| 字段级类型（VARCHAR/INT） | ❌ 下沉到 LLD |
| 单接口路径与参数 | ❌ 下沉到 LLD |
| 像素/颜色/CSS | ❌ 设计文档不关心 UI 视觉，那是原型职责 |

## 五、质量清单

每个 HLD 章节写完后对照：

- [ ] 系统边界清晰（负责/不负责/外部交互）
- [ ] 有系统总体架构图且有文字解读
- [ ] 分层设计完整，每层有职责和技术
- [ ] 功能模块清单每一行都标注了对应 SRS 功能编号（可追溯）
- [ ] 模块依赖关系明确，无未说明的循环依赖
- [ ] 核心实体清单覆盖 SRS 数据需求，有概念 ER 图
- [ ] 存储选型有划分依据
- [ ] 接口架构按分组说明，未下沉到单接口
- [ ] 技术选型每项有理由
- [ ] 非功能需求逐条映射到架构策略
- [ ] 无越界内容（无字段级表结构、无单接口明细、无代码）
