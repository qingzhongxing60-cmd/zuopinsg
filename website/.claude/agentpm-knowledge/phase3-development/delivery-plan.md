# 项目交付计划规范

## 核心原则

交付计划是**项目级的唯一真相**，记录从前端原型到上线的完整交付链路。所有技能（page-generator、未来的 backend-generator 等）都通过读写同一个计划文件来协作，不依赖会话上下文。

---

## 计划文件位置与命名

```
docs/delivery-plan.md
```

固定路径，不随项目名变化。每个项目只有一个交付计划文件。

---

## Phase 分层规范

交付计划按阶段（Phase）组织，每个 Phase 对应一组技能：

| Phase | 名称 | 对应技能 | 说明 |
|-------|------|---------|------|
| Phase 1 | 前端原型 | page-generator | Mock 数据驱动，无真实后端 |
| Phase 2 | 后端开发 | backend-generator（待建） | API、数据库、业务逻辑 |
| Phase 3 | 前后端联调 | api-connector（待建） | 替换 Mock 为真实接口 |
| Phase 4 | 测试与上线 | pm-test-cases + test-runner（待建） | 测试执行、部署 |

**扩展规则**：新增 Phase 只需在计划文件末尾追加，不修改已有 Phase。

---

## Phase 1 功能分层规则

Phase 1（前端原型）内部按功能依赖关系分层，决定实现顺序。**分层的本质是：一个功能能不能独立运行，不依赖其他功能的数据或接口。越早能独立运行，层级越低。**

### 第1层：入口与基础框架（最先实现）

包含以下类型，无论 PC 端还是移动端：
- **登录/登出页面**：所有其他功能的前提，必须最先实现
- **移动端"我的"页面**：个人信息、设置、退出登录，无业务依赖
- **系统管理类**：用户管理、角色权限、系统参数配置、操作日志

### 第2层：基础数据管理

- 所有主数据管理模块（炼厂、承运商、车辆、司机、油库等）
- 判断标准：不依赖业务流程数据，只依赖第1层的系统框架

### 第3层：核心业务 + 移动端工作台

- PC端主要业务流程模块（运单管理、在途监控、接卸管理等）
- **移动端工作台各应用**（创建运单、运单管理、在途监控、接卸作业等）
- **关键**：移动端工作台功能与PC端核心业务同层，不要推到最后

### 第4层：衍生功能 + 移动端首页/消息

- PC端：预警管理、报表统计（需要大量业务数据才有意义）
- 移动端：首页（需要运单和接卸数据做概览）、消息（需要预警数据）

### 移动端分层规则（重要）

移动端功能**不能全部放到最后**，要按实际依赖关系分层：

| 移动端功能类型 | 应放层级 |
|-------------|---------|
| 登录/登出 | 第1层 |
| 我的（个人信息+设置） | 第1层 |
| 工作台各应用（创建运单/运单管理/接卸等） | 第3层 |
| 首页（待办+数据概览） | 第4层 |
| 消息（通知+预警） | 第4层 |

### 已有页面处理

生成计划前先扫描项目现有代码，已有的页面不纳入待实现计划，在备注列注明"已有，跳过"。

## 状态标识规范

| 标识 | 含义 |
|------|------|
| ⬜ | 待实现 |
| 🔄 | 进行中（当前会话正在生成） |
| ✅ | 已完成（page-generator 验收通过） |
| ⏸ | 暂停（依赖未就绪或人工暂停） |
| ❌ | 跳过（不在本期实现范围） |

---

## ID 命名规范

- Phase 1 前端：`F01`、`F02`...（F = Frontend）
- Phase 2 后端：`B01`、`B02`...（B = Backend）
- Phase 3 联调：`C01`、`C02`...（C = Connect）
- Phase 4 测试：`T01`、`T02`...（T = Test）

同一功能在不同 Phase 使用不同 ID，通过功能名关联（如 F01 用户管理 → B01 用户管理 API）。

---

## 更新规则

**技能完成后必须更新计划文件：**
1. 将对应条目状态改为 ✅
2. 填写完成时间（当天日期）
3. 更新总进度表中的计数
4. 更新"最后更新"日期
5. 更新"当前推荐"为下一个可实现的功能（依赖已满足且状态为 ⬜）

**推荐下一步的逻辑：**
- 优先推荐同层级中依赖已全部满足的功能
- 同层级多个可选时，按 ID 顺序推荐
- 当前层级全部完成后，推荐下一层级的第一个功能
