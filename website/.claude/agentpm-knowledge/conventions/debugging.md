# 系统化调试

## 核心原则

```
没有根因调查就提出修复 = 猜测，不是调试
```

随机修复浪费时间并引入新 bug。快速补丁掩盖底层问题。

---

## 铁律

```
没有完成阶段 1，不能提出修复方案
```

---

## 何时使用

用于任何技术问题：测试失败、生产 bug、意外行为、性能问题、构建失败、集成问题。

**尤其在以下情况必须使用：**
- 时间压力下（紧急情况让猜测变得诱人）
- "一个快速修复"看起来很明显
- 已经尝试了多次修复
- 上一次修复没有生效

---

## 四阶段流程

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息** — 不要跳过，读完整堆栈
2. **稳定复现** — 能可靠触发吗？具体步骤是什么？
3. **检查最近变更** — git diff、最近提交、新依赖、配置变更
4. **多组件系统：在边界处收集证据**
   - 对每个组件边界：记录进入数据、退出数据
   - 运行一次收集证据，确定哪个层级出问题
   - 然后只调查那个具体组件
5. **追踪数据流** — 坏值从哪里来？谁传入的？一直追溯到源头

### 阶段 2：模式分析

1. **找到正常工作的类似代码** — 同代码库中类似的实现
2. **对比差异** — 列出所有差异，不要假设"这个不重要"
3. **理解依赖** — 需要什么组件、配置、环境？

### 阶段 3：假设与测试

1. **形成单一假设** — "我认为 X 是根因，因为 Y"
2. **最小化测试** — 做最小可能的改动来验证假设
3. **一次一个变量** — 不要同时修多个东西
4. **无效就回滚** — 不要在失败的修复上叠加更多修复

### 阶段 4：实现修复

1. **创建失败测试** — 最简复现，自动化测试
2. **实现单一修复** — 只修根因，不做"顺手"改进
3. **验证修复** — 测试通过？其他测试没坏？
4. **如果修复无效** — 见下方三次失败规则

---

## 三次失败规则

**如果同一个方向尝试 3 次仍然失败：**

```
停止修补。质疑架构。
```

### 架构问题的信号

- 每次修复揭示新的共享状态/耦合问题
- 修复需要"大规模重构"才能实现
- 每次修复在别处产生新症状

### 处理方式

1. 停止尝试第 4 次修复
2. 退后一步，质疑基本假设
3. 这个模式本身是否合理？
4. 是否应该重构架构而非继续修症状？
5. **向用户说明情况，讨论后再行动**

---

## Red Flags — 立即停止并回到阶段 1

如果你发现自己在想：
- "先快速修一下，以后再调查"
- "试试改 X 看看行不行"
- "同时改多个地方，跑测试"
- "跳过测试，手动验证"
- "大概是 X，让我修一下"
- "我不完全理解但这可能有效"
- "再试一次修复"（已经试了 2+ 次）
- 在追踪数据流之前就提出解决方案

**所有这些都意味着：停止。回到阶段 1。**

## 反借口表

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因，流程对简单 bug 很快 |
| "紧急情况，没时间走流程" | 系统化调试比猜测-检查-再猜更快 |
| "先试这个，再调查" | 第一次修复定下模式，从一开始就做对 |
| "修好后再写测试" | 未测试的修复不持久，先测试证明它 |
| "同时修多个省时间" | 无法隔离哪个有效，还会引入新 bug |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因 |
| "再试一次"（2+ 次失败后） | 3+ 次失败 = 架构问题，别再修了 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| 1. 根因 | 读错误、复现、查变更、收集证据 | 理解 WHAT 和 WHY |
| 2. 模式 | 找正常例子、对比 | 识别差异 |
| 3. 假设 | 形成理论、最小测试 | 确认或新假设 |
| 4. 实现 | 创建测试、修复、验证 | Bug 解决，测试通过 |
