# Git 工作流

## 提交消息格式

```
<类型>(<范围>): <描述>

<可选正文>
```

| 类型 | 说明 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复缺陷 |
| `refactor` | 代码重构（不含 bug 修复和功能新增） |
| `docs` | 文档变更 |
| `test` | 添加或修改测试 |
| `chore` | 构建工具或辅助库变更 |
| `perf` | 性能优化 |
| `ci` | CI 配置变更 |

示例：
```
feat(auth): 新增手机号登录功能
fix(upload): 修复大文件上传超时问题
refactor(user): 拆分用户服务为独立模块
```

## 分支管理

- 主分支：`main`（生产）、`dev`（开发）
- 功能分支：`feat/功能名`
- 修复分支：`fix/问题描述`
- 永远不要直接推送到 `main`

## PR 工作流

创建 PR 时：
1. 分析完整提交历史（不只是最新提交）
2. 用 `git diff [base-branch]...HEAD` 查看所有变更
3. 写清晰的 PR 描述（做了什么、为什么、如何测试）
4. 新分支用 `-u` 标志推送：`git push -u origin feat/xxx`

## 提交前检查

- [ ] 代码已通过本地测试
- [ ] 无 `console.log` 遗留
- [ ] 无硬编码密钥或配置
- [ ] 提交消息符合格式规范
