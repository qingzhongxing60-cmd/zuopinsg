# 代码审查规范

## 两阶段审查流程

**阶段 1：需求符合性审查**
- 对照需求/设计文档，逐项检查功能是否完整实现
- 检查边界场景是否处理
- 检查交互行为是否与需求一致

**阶段 2：代码质量审查**
- 安全性、性能、可维护性
- 遵循下方检查清单

两阶段独立执行，不能合并。阶段 1 发现缺失功能时，修复后必须重新执行阶段 2。

## 必须审查的场景

- 编写或修改代码后
- 提交到共享分支前
- 涉及安全敏感代码（认证、支付、用户数据）
- 架构变更
- 合并 PR 前

## 审查检查清单

标记完成前必须确认：

- [ ] 代码可读，命名清晰
- [ ] 函数职责单一（< 50 行）
- [ ] 文件聚焦（< 800 行）
- [ ] 无深层嵌套（> 4 层）
- [ ] 错误处理完整
- [ ] 无硬编码密钥或配置
- [ ] 无 `console.log` 或调试语句
- [ ] 新功能有对应测试
- [ ] 测试覆盖率 ≥ 80%

## 安全审查触发点

遇到以下情况必须停下来做安全审查：

- 认证/授权代码
- 用户输入处理
- 数据库查询
- 文件系统操作
- 外部 API 调用
- 加密操作
- 支付/金融代码

## 问题严重级别

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| CRITICAL | 安全漏洞或数据丢失风险 | **阻断** — 必须修复后才能合并 |
| HIGH | Bug 或重大质量问题 | **警告** — 应在合并前修复 |
| MEDIUM | 可维护性问题 | **提示** — 建议修复 |
| LOW | 风格或小建议 | **备注** — 可选 |

## 常见问题

### 安全类
- 硬编码凭证（API Key、密码、Token）
- SQL 注入（查询中字符串拼接）
- XSS 漏洞（未转义用户输入）
- 路径遍历（未净化文件路径）
- 缺少 CSRF 保护
- 认证绕过

### 代码质量类
- 超长函数（> 50 行）→ 拆分
- 超大文件（> 800 行）→ 提取模块
- 深层嵌套（> 4 层）→ 提前返回
- 缺少错误处理
- 直接修改对象（应用不可变模式）
- 缺少测试

### 性能类
- N+1 查询 → 用 JOIN 或批量查询
- 缺少分页 → 加 LIMIT
- 无限制查询 → 加约束条件
- 缺少缓存 → 对高频查询加缓存

## 审查通过标准

- **通过**：无 CRITICAL 或 HIGH 问题
- **谨慎合并**：仅有 HIGH 问题
- **阻断**：发现 CRITICAL 问题

## 常见误报（跳过这些）

以下情况不应报告为问题：

- "考虑添加错误处理" — 调用方或框架已处理错误路径时
- "缺少输入验证" — 函数是内部的且调用方已验证时
- "魔法数字" — 对于 200、404、1000ms、60、24 等众所周知的常量
- "函数太长" — 对于穷举的 switch 语句、配置对象、测试表
- "缺少 JSDoc" — 对于名称和签名已自描述的单用途内部辅助函数
- "可能的空引用" — 前一行已收窄类型或有 if 守卫时
- Mock 数据中的测试用密码/token — 不算安全问题

**置信度规则**：80% 以上确信才报告。不确定时降级或放弃。
