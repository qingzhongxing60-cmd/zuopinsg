# 开发工作流

## 铁律

```
没有验证证据的完成声明 = 谎言
没有 code-reviewer 的代码提交 = 违规
没有技能检查的任务执行 = 违规
```

## 流程（每一步都是强制的）

1. **接收任务**
2. **技能检查** — 扫描触发词，1% 可能性就调用 Skill 工具
3. **实现** — 规范由 hook 自动注入，直接写代码
4. **code-reviewer** — 写完代码立即调用，强制，无例外（涵盖代码质量+安全）
5. **验证** — 运行构建/测试命令，看到输出后才能声明完成
6. **报告结果** — 只有 step 4-5 全部通过后才能向用户报告

## 验证铁律

```
完成声明前必须有新鲜的验证证据。
如果你没有在当前消息中运行验证命令，你不能声称它通过了。
```

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| "测试通过" | 测试命令输出：0 failures | 之前的运行、"应该通过" |
| "构建成功" | 构建命令：exit 0 | linter 通过、日志看起来正常 |
| "Bug 已修复" | 复现原始症状：通过 | 代码改了、假设修好了 |
| "需求已满足" | 逐条对照检查清单 | 测试通过 |
